1 # OpenSSL Engine: TLS example {#page_a71ch_opensslengine_tls}
2 
3 - DocRevision : 0.92
4 - Date : 2018-04-04
5 
6 This section explains how to set-up a TLS demo using the A71CH OpenSSL Engine.
7 (Consult \ref page_imx6ul_setup_openssl_engine for information on how to prepare the
8 embedded platform.)
9 
10 # (1) Summary
11 
12 The TLS demo demonstrates setting up a mutually authenticated and
13 encrypted link between a client and a server system.
14 The keypair used to identify the client is stored in the Secure Element.
15 The keypair used to identify the server is simply available as a pem file.
16 
17 The public keys associated with the respective key pairs are contained in respectively a client and a
18 server certificate.
19 
20 The signing (CA) keypair and associated certificate is a self-signed certificate.
21 The same CA is used to sign client and server certificate.
22 
23 The TLS demo comes in two flavours:
24 
25 1. \b Flavour-A: The standard OpenSSL tools \p s_server and \p s_client are used to set-up and demonstrate the TLS link. The certificates used are simply stored on the file system of the host. The client uses the keypair stored inside the A71CH.
26 2. \b Flavour-B: A TLS client program - included in source code (a71chTlsClient.c) - retrieves the client certificate from the A71CH and uses the keypair stored inside the A71CH. It establishes a TLS connection with the server process \p s_server.
27 
28 Steps (2) to (5) are identical for the two demo flavours.
29 
30 # (2) Credential preparation (execute once) [Optional]
31 
32  > cd axHostSw/hostLib/embSeEngine/a71chDemo/scripts
33  > ./tlsCreateCredentialsRunOnClientOnce.sh
34 
35 \note The Host SW package comes bundled with the
36 required credentials. The \p tlsCreateCredentialsRunOnClientOnce.sh script will re-create equivalent credentials
37 (but with new/different keypairs)
38 
39 To simplify the credential creation process the script creates both client and server credentials (once)
40 on the client platform.
41 One must transfer the server credentials created to the server platform.
42 
43 # (3) Secure Element preparation (client side)
44 
45 For the purpose of the demo one MUST inject the TLS client key pair into the Secure Element
46 and create a reference pem file referring to this provisioned key pair.
47 
48  > cd axHostSw/hostLib/embSeEngine/a71chDemo/scripts
49  > ./tlsPrepareClient.sh
50 
51 # (4) Server side preparation
52 
53 \note The Host SW package comes bundled with the required server credentials.
54 
55 Transfer the default server credentials or those created under (2) to the server platform.
56 
57 # (5) Start up the server
58 
59 \note The server can run e.g. on a PC. The server must be reacheable
60 over the TCP/IP network for the Client.
61 
62 Execute the following command on the server platform, make a note on the IP address of the server.
63 
64  > cd axHostSw/hostLib/embSeEngine/a71chDemo/scripts
65  > ./tlsServer.sh [ECDHE|ECDH|both|max]
66 
67 # (6) Establish a TLS link from the client to the server
68 
69 The client process establishing the TLS connection comes in two flavours: either \p s_client or a program provided in source code (a71chTlsClient.c).
70 Invoke either example through a bash shell script.
71 
72 ## (6.1) Using s_client
73 
74 Invoke the script using the IP address of the server as the first argument and ECDH or ECDHE as the second argument (ECDHE corresponding to ECDH ephemeral).
75 
76  > ./tlsSeClient.sh <server-IP-address> [ECDH|ECDHE]
77 
78 ## (6.2) Using a71chTlsClient.c
79 
80 Invoke the script using the IP address of the server as argument.
81 
82  > ./a71chTlsClient.sh <server-IP-address>
83
